## 修复 sniffer 漏掉 Message Counter 字节导致解析失败

- **根因**：`crid_sniffer.c` 在提取 Vendor Specific IE 数据时只跳过了 `ID + Len + OUI(3) + OUI_Type(1) = 6` 字节，但发送端（`main_tx/crid_messages.c`）的格式是 `OUI(3) + VendorType(1) + **MessageCounter(1)** + PackedMsg`，即 OUI Type 之后还有一个 Counter 字节。
- **修复**：
  - `data_offset`：`offset + 6` → `offset + 7`
  - `vendor_data_len`：`ie_length - 4` → `ie_length - 5`
  - 同时更新了非 RID Vendor IE 采样路径和 `crid_parser.c` 的注释
- **影响**：之前 data[0] 拿到的是 Counter 字节（0-255），被当作 Message Pack header 解析自然失败；现在 data[0] 正确指向 Packed Message header (0xF2) 或单消息头。

## 重构 parser/sniffer：利用 opendroneid 的 ODID_service_info 结构体

- **之前**：sniffer 手动跳过 Counter 字节只传 Message Pack，parser 手写 `try_decode()` 判断 header
- **现在**：
  - Sniffer 传递 `ODID_service_info` 格式数据（即 OUI_Type 之后的 `[Counter + Pack]`），`data_offset = offset + 6`
  - Parser 直接使用 `ODID_service_info *` 指针，调用 `odid_message_process_pack()` 解析（来自 opendroneid wifi 库）
  - 删除了手写的 `try_decode()` 函数
  - `odid_message_process_pack()` 内部会调用 `odid_initUasData()` + `decodeMessagePack()`，更标准
  - 引入 `#include "odid_wifi.h"`
- **保留**：单消息 fallback 策略（`decodeOpenDroneID`）仍保留用于非 Packed 格式

## 国标（GB 42590-2023）Message Pack 解析位置修复

- **问题**：国标格式在 `message_counter` 之后有 3 字节管理信息（`0xF1` + `SingleMsgSize` + `MsgCount`），然后才是消息体。ASTM 格式中 `message_counter` 后直接是 Message Pack（首字节 `0xF2`）。parser 之前统一按 ASTM 格式解析，导致国标消息偏移 3 字节。
- **修复**：`crid_parser_decode()` 新增策略 2 —— 检测 `data[1] == 0xF1`（国标标识），跳过 3 字节管理信息后构造 ASTM 兼容的 `ODID_MessagePack_encoded` 头部再调用 `odid_message_process_pack()` 解析。策略 1（ASTM 0xF2）和策略 3（单消息 fallback）不变。
