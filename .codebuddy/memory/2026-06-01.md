## 双 OUI 标准支持修复（ASTM + GB 42590）
- **根因**：树莓派 spoofing 代码使用 ASTM F3411 国际标准 OUI=`FF:FF:5F`，而 ESP32 RX 代码硬编码只匹配 GB 42590 的 OUI=`FA:0B:BC`，导致树莓派信号完全被忽略
- **修复**：
  - `crid_rx_types.h`：新增 `OUI_ASTM_0/1/2` (FF:FF:5F) 和 `OUI_CRID_0/1/2` (FA:0B:BC)，添加宏 `IS_RID_OUI(o0,o1,o2)` 同时匹配两种 OUI
  - `crid_sniffer.c`：用 `IS_RID_OUI` 替换原来的硬编码匹配，并根据 OUI 类型自动选择数据偏移量：
    - ASTM F3411: `offset+6`（无 Counter 字节），`data_len = ie_length - 4`
    - GB 42590: `offset+7`（有 Counter 字节），`data_len = ie_length - 5`
- **同时**：保留了上一轮的 Beacon/非 RID Vendor IE 诊断采样功能

- **根因再分析**：参考能正常工作的 `esp32_crid_scan.c`，发现它统一使用 `offset+6` 提取数据，依赖 `decodeMessageType(data[0])` 动态判断。树莓派 GB 42590 格式是 `[Counter][Reserved/0x00][PackHeader(0xF2)][MsgSize][MsgCount][messages]`，跳过 Counter 后 data[0]=0x00 被误判为 BASIC_ID 导致解码失败
- **最终修复**：
  - `crid_sniffer.c`：统一使用 `offset+6`（不区分 ASTM/GB），让 parser 自适应
  - `crid_parser.c`：3 步自适应解析：①直接以 data[0] 解析（ASTM）②跳过 1 字节（Counter）③跳过 2 字节（Counter+Reserved）
  - 同时保留双 OUI 支持（ASTM `FF:FF:5F` + GB `FA:0B:BC`）

## 关闭 SSID 诊断输出 & 修复 Message Pack 误判

  - 移除 Non-RID Vendor IE 的去重采样和每 60 秒打印

  - 清理相关的 `static` 变量声明

- **解析错误修复**（`crid_parser.c`）：
  - **现象**：`Message Pack decode failed (err=1) SingleMsgSize=242 MsgPackSize=25`
  - **根因**：策略 2 检查 `data[1]` 高 4 位是否为 `0xF`（PACKED header），但 `data[1]` 恰好为 `0xF2`（=242）的字节被误判为 PACKED header。`decodeMessagePack` 中 `SingleMsgSize=242` 不合法（应为 25），导致解码失败
  - **修复**：`try_decode()` 的 PACKED 分支中增加快速验证：`pack->SingleMessageSize != ODID_MESSAGE_SIZE` 时直接返回 `ODID_FAIL`，避免进入 `decodeMessagePack` 产生告警日志

## 借鉴 open-remote-id-parser (ORIP) 重构数据结构分层

参考 https://github.com/iannil/open-remote-id-parser 的设计：
- ORIP 是 C++17 跨平台库，支持 ASTM/ASD-STAN/GB 三协议、BT/WiFi 双传输、异常检测/轨迹分析

### 第 1 项：完善枚举和结构体（`crid_rx_types.h`）
- 新增 `rid_protocol_t` 枚举：UNKNOWN / ASTM_F3411 / ASD_STAN / CN_RID
- 新增 `cn_uav_category_t` 枚举（GB 42590 中国无人机分类）：微型/轻型/小型/中型/大型（按重量 5 级）
- 新增 `cn_flight_zone_t` 枚举（GB 42590 飞行区域）：适飞/限制/禁飞空域
- 新增 5 个分层结构体，与 ODID_UAS_Data 解耦：
  - `rid_location_t`：位置/速度/方向/精度（14 字段 + valid 标志）
  - `rid_system_info_t`：操作员位置/区域/EU 分类（11 字段 + valid 标志）
  - `rid_self_id_t`：Self-ID 描述文本
  - `rid_operator_id_t`：操作员 ID
  - `rid_basic_id_t`：Basic ID 信息

### 第 3 项：数据结构分层（uav_track_t 扩展）

- 保留底层 `ODID_UAS_Data` + `ODID_MessagePack_data` 不变

### 解析器更新（`crid_parser.c/h`）
